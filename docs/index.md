# Python Virtual Environment Manager Documentation

## Self-Healing Logic

### The Problem
When a project folder containing a virtual environment is moved or renamed, the existing `.venv` can become "broken." This happens because the `activate` script generated by `venv` (or `virtualenv`) contains hardcoded absolute paths to the original location.

Even if the Python executable (which is often a symlink) still seems functional, the `PATH` within the activated environment will point to non-existent directories. This can cause issues like:
- `pip` falling back to the system version, which may fail if a global `pip.conf` requires an active virtual environment (e.g., `ERROR: Could not find an activated virtualenv (required).`).
- Shebang lines in installed scripts pointing to incorrect paths.

### The Solution: Self-Healing Logic
The `venv_manager.sh` script includes an improved self-healing mechanism to handle these scenarios:

1. **Detect Moves**: It compares the actual current path of the `.venv` directory with the `VIRTUAL_ENV` path defined inside the `bin/activate` script.
2. **Automatic Rebuild**: If a mismatch is detected, the script assumes the folder has been moved. It automatically deletes the broken `.venv` and rebuilds it from scratch.
3. **Restore Functionality**: By rebuilding the environment, all internal paths and shebangs are updated to the new location, ensuring tools like `pip` and other project dependencies work correctly.
